# Token Explorer

## 这个应用是什么

Token Explorer 是一个逐步拆解大语言模型"选词过程"的交互式工具。

普通聊天界面的流程是：你输入一句话，模型返回一整段回复，中间的过程完全不可见。Token Explorer 把这个过程彻底打开——每次只往前走一步，展示模型在这一步看到的所有候选词及其概率，然后由你来决定选哪个词继续。

它的核心定位是**探索和学习**，不是高效生成文本。

## 为什么做这个

学习大语言模型时，最根本的概念是 next token prediction：模型每次只预测"下一个 token 是什么"，通过不断重复这个动作来生成文本。但这个概念在实际产品中完全被隐藏了——你只看到最终结果，看不到每一步的决策过程。

我们想要一个工具，让你能：

- 亲眼看到模型在每一步给出的完整概率分布
- 直观感受 temperature 参数如何改变这个分布的形状
- 通过手动选择不同的 token，观察它如何影响后续预测
- 在 token 级别（而非字符级别）理解文本的构成

## 核心交互流程

1. 你在编辑器中输入文本
2. 后端对文本进行 tokenize（分词），前端实时高亮显示每个 token 的边界
3. 后端同时运行模型的前向传播，计算"下一个 token"的概率分布
4. 左侧列表展示概率最高的候选 token，按概率从高到低排列
5. 你点击某个候选 token，它被追加到编辑器中
6. 重复上述过程——新的文本触发新的预测

整个过程中，模型永远不会自动选择任何 token。每一步都等你来决定。

## 编辑器的设计意图

编辑器不是普通的文本框，而是一个 **token 级别的编辑器**。这是这个项目中最特别的设计决策之一。

### Token 级别操作

- **删除**：按退格键不是删除一个字符，而是删除光标前的整个 token。因为 LLM 不理解"字符"，它的最小单位就是 token
- **光标移动**：左右方向键在 token 之间跳转，而不是在字符之间移动
- **视觉高亮**：每个 token 用不同的背景色标记，类似于在线 tokenizer 可视化工具的效果。8 种颜色循环使用，相邻 token 用不同颜色以便区分边界

### 双重渲染机制

编辑器内部有两种渲染模式：

- **已同步模式**：当 tokenize 的结果和当前文本完全匹配时，显示彩色高亮的 token 视图
- **未同步模式**：在用户快速输入、tokenize 结果尚未返回时，显示纯文本视图

之所以需要两种模式，是因为 tokenize 需要网络请求（约 80 毫秒的防抖延迟），如果每次输入都等 tokenize 完成再渲染，打字会明显卡顿。双重渲染保证了无论 tokenize 是否完成，编辑器始终流畅响应。

两种模式的切换必须是无缝的——它们占用完全相同的空间（token 高亮不增加任何额外的内边距或边框宽度），所以切换时不会产生文字跳动。

## 可调参数

### System Prompt（系统提示词）

给模型设定角色或指令。当系统提示词为空时，模型以"文本补全"模式运行；当有系统提示词时，模型以"对话"模式运行，你的输入会被包裹为 user 消息，模型会以 assistant 身份生成回复。

### Temperature（温度）

控制概率分布的"锐利程度"。

- 设为 0 时，最高概率的 token 几乎独占 100%，其它候选概率趋近于零。模型极度确定，每次选同一个词
- 设为 1 时，保持模型的原始概率分布不变
- 设为 2 时，概率分布被大幅"压平"，原本概率低的 token 获得更多机会，分布变得更均匀、更随机

Temperature 的本质是在 softmax 之前对原始得分（logits）除以这个值。除以一个大于 1 的数会让所有得分更接近，除以一个小于 1 的数会放大差异。

在 Explorer 中调节 temperature，你可以实时观察候选列表的概率如何变化——这是理解这个参数最直观的方式。

### Top-K（候选数量）

控制候选列表中展示多少个 token。默认为 20，意味着只显示概率最高的前 20 个候选。

这个参数**仅影响展示数量**，不影响模型的计算过程。模型始终对整个词汇表（约 15 万个 token）计算概率分布，Top-K 只是从中截取前 K 个给你看。

## 推理信息

每次预测完成后，侧边栏会显示当次推理的技术细节：

- **Prefill**：首次处理整段文本所花的时间，以及处理速度（token/秒）
- **Step**：在已有缓存基础上处理新增 token 所花的时间（通常比 Prefill 快得多）
- **Cache**：当文本没有变化但参数变了时，会显示"Reused"——因为模型的原始输出（logits）被缓存了，只需要重新应用 temperature 即可，不需要重新推理
- **Sequence**：当前文本的总 token 数
- **Vocab**：模型词汇表大小

## KV 缓存复用

这是一个重要的性能优化。大语言模型处理文本时会产生"键值缓存"（KV Cache），记录已处理 token 的中间计算结果。

当你追加一个 token 时，后端不需要重新处理整段文本——它会检测新文本和已缓存文本的公共前缀，只处理新增的部分。这就是为什么第一次输入可能需要几百毫秒（Prefill），但后续追加 token 只需要几十毫秒（Step）。

如果你删除了中间的 token，或者修改了前面的内容，缓存会失效，需要重新做完整的 Prefill。

## 快捷键

- **点击候选 token**：追加到编辑器并触发下一次预测
- **数字键 1-9**（编辑器失焦时）：快速选择对应排名的候选 token
- **Enter**（编辑器失焦时）：选择排名第一的候选 token
- **⌘+Enter**：立即触发预测（跳过防抖等待）
- **⌘+Backspace**：清空全部文本
- **Escape**：编辑器失焦

## 本地运行

模型通过 Apple 的 MLX 框架运行在本地 Apple Silicon 上。模型在后端服务启动时加载到内存，并一直保持在内存中直到服务关闭，避免反复加载的时间开销。所有推理过程都在本地完成，不需要网络连接。
